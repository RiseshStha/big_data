  0: 1. Introduction
  1: In the contemporary digital landscape, data generation has scaled exponentially, necessitating the adoption of specialised distributed computing systems capable of processing, storing, and analysing information at unprecedented scales. This academic report investigates the StackOverflow StackSample dataset through the comprehensive five-dimensional framework of big data, commonly referred to as the five Vs: Volume, Variety, Velocity, Veracity, and Value.
  2: Table 1: The Five Vs of Big Data Applied to the Stack Overflow StackSample Dataset
  3: 
  4: Processing long-form unstructured HTML text at this scale poses significant computational challenges that cannot be addressed by conventional single-node analytical tools. Standard packages such as Python Pandas rely entirely on single-threaded, in-memory processing. When subjected to over 1.26 million rows of long-form text and the subsequent generation of high-dimensional TF-IDF feature matrices, single-node architectures inevitably suffer JVM heap space exhaustion or system-wide memory failure due to the absence of lazy evaluation, horizontal scaling, and distributed partition management. Consequently, Apache PySpark is utilised as the essential technological framework to enable distributed in-memory processing through resilient distributed datasets and partition-based parallel execution across all available CPU cores.
  5: 
  6: This investigation pursues three primary analytical objectives. First, the study implements multi-class tag classification using a Random Forest model to predict the primary technology tag of a question from its text content alone. Second, it executes quality score prediction using Linear Regression to examine whether linguistic features can explain the community-voted score of a question. Third, it conducts unsupervised Latent Dirichlet Allocation to discover the underlying thematic structure of the corpus. Tableau is utilised exclusively as the visualisation platform to construct six interactive dashboards, transforming processed PySpark outputs into high-fidelity analytical representations. Table 2 presents a consolidated summary of all results across the four analytical phases.

 12: 2. Implementation
 13: 2.1 Dataset Description
 14: The empirical research utilises the public StackSample dataset, a representative 10% subset of the official Stack Exchange Data Dump. This dataset is published on Kaggle by Stack Overflow under a Creative Commons Attribution-ShareAlike licence and is freely accessible at the following direct link: https://www.kaggle.com/datasets/stackoverflow/stacksample (Kaggle, 2019). The data is organised into three separate comma-separated value files: Questions.csv containing 1,264,216 rows and 7 columns, Answers.csv containing 2,014,516 rows, and Tags.csv containing 3,750,994 rows and 2 columns. For the scope of this project, Questions.csv is inner-joined with Tags.csv on the primary key identifier to map technology tag categories to their corresponding question text.
 15: 
 16: The analytical pipeline processes five distinct data types, satisfying rigorous data variety requirements. These types comprise integer identifiers and scores (Id, Score), long-form variable-character HTML text (Body), short-character title strings (Title), temporal timestamps (CreationDate), and categorical tag strings (Tag).
 17: 
 18: To ensure the downstream machine learning models receive clean, high-quality numerical inputs, several preprocessing stages are executed sequentially within the PySpark distributed pipeline. First, a regular expression replacement function is applied using Spark SQL functions, substituting HTML elements matching the pattern <[^>]+> with a single whitespace character to extract clean, unformatted text. Second, null value filtering is performed; this step confirms that zero null values exist within the critical columns of Title, Body, and Score, validating the completeness of the dataset. Third, the cleaned Title and Body strings are concatenated into a singular unified column named text to create a consistent NLP input corpus. Fourth, a logarithmic transformation represented as log1p(Score) is applied to compress the highly skewed question score distribution for the regression stage. Fifth, the Tags.csv file is aggregated using the collect_list and concat_ws functions, grouping all technology tags associated with a specific question into a space-delimited string which is subsequently inner-joined to the main questions schema.
 19: 2.2 Environment and Configuration
 20: The distributed computing environment is established on a Windows 11 platform, executing Apache Spark version 3.5.8 integrated with Python version 3.10. The SparkSession is configured with a driver memory allocation of 4 gigabytes and a shuffle partition count of 50 to 100, optimised for local multi-core parallel execution.
 21: 
 22: Apache Spark operates on a centralised master-worker driver-executor architecture. The driver program executes the main application code, maintains the SparkContext, translates user operations into directed acyclic graphs of execution stages, and schedules tasks across the available executor processes. The executors are independent worker processes assigned to specific CPU cores that perform partition-level computations in parallel and temporarily store intermediate results in physical memory. Distributed in-memory computation is absolutely critical for this project. Given that the text column alone constitutes several gigabytes of raw string data, standard single-node systems would exhaust available physical memory when constructing high-dimensional TF-IDF vectors across 1.26 million rows. PySpark bypasses this limitation by partitioning the dataset into logical data blocks and processing each partition concurrently across all available processor cores.
 23: 2.3 Task 1: Tag Classification (Random Forest)
 24: The machine learning pipeline for multi-class classification is constructed using PySpark MLlib through a six-stage Natural Language Processing pipeline that transforms raw question text into predictive numerical features. The Tokenizer first converts the unified text column into lists of lowercase word tokens. The StopWordsRemover then filters standard English stop words that carry minimal semantic discriminative value, such as pronouns, prepositions, and auxiliary verbs. HashingTF subsequently maps filtered tokens into a high-dimensional sparse vector using a feature hash space of 32,768 dimensions, encoding term frequency counts for each word. The IDF estimator then scales raw term frequencies by their inverse document frequency, penalising terms that appear ubiquitously across the entire corpus and thereby amplifying the discriminative power of domain-specific technical vocabulary. The StringIndexer encodes the categorical target variable primary_tag as a numerical label. Finally, a Random Forest Classifier with 50 decision trees, a maximum depth of 10, and a subsampling rate of 0.8 is trained on the resulting feature vectors.
 25: 
 26: To maintain computational feasibility on a local machine while preventing class imbalance, stratified sampling is implemented to extract approximately 20,000 records per tag class. The resulting balanced sample of 200,074 rows is partitioned into an 80% training set of 160,236 rows and a 20% test set of 39,838 rows. The top ten technology tags selected are python, javascript, java, c#, php, android, html, jquery, c++, and ios.
 27: 
 28: The Random Forest algorithm is selected over multinomial Logistic Regression for several theoretical and practical reasons. It naturally handles high-dimensional sparse TF-IDF vectors without requiring feature scaling or regularisation tuning. It inherently supports multi-class classification without one-versus-rest decomposition. It is more robust to overfitting on noisy text data through decision tree ensemble averaging and feature bootstrapping. Furthermore, it produces native feature importance scores enabling post-hoc interpretability of the most predictive vocabulary terms.
 29: 
 30: The classification model achieved an overall accuracy of 71.31%, an F1-score of 0.7076, a weighted precision of 0.7204, and a weighted recall of 0.7131, completing training in 6 minutes and 35 seconds. Table 3 presents the per-class accuracy rates across all ten technology tags.

 39: 2.4 Task 2 — Score Prediction (Linear Regression)
 40: The second analytical task employs a text-based regression pipeline to predict StackOverflow question quality scores. The pipeline mirrors the initial NLP stages of the classification pipeline, utilising tokenization, stop word removal, HashingTF with 16,384 features, and IDF scaling. A VectorAssembler subsequently concatenates the TF-IDF sparse feature vector with a numerical text_length feature representing question character count. The consolidated feature vector is passed to a Linear Regression estimator configured with a maximum iteration count of 10, a regularisation parameter of 0.1, and an ElasticNet mixing parameter of 0.5 to balance L1 and L2 weight penalties.
 41: 
 42: The target variable, community Score, exhibits a severely right-skewed distribution, with a global mean of 1.78, a minimum of -73, and a maximum of 5,190. A log1p transformation is applied prior to modelling to compress the distribution, stabilise residual variance, and prevent extreme outlier scores from disproportionately influencing regression coefficients. The model is trained on a stratified sample of 100,000 positive-score questions (Score ≥ 0), partitioned into approximately 80,000 training rows and 20,000 test rows.
 43: 
 44: Linear Regression is selected for this task as it provides highly interpretable coefficients and trains rapidly on the sampled dataset, establishing an appropriate baseline to evaluate whether linear relationships exist between textual characteristics and community-voted scores. The regression model achieved an RMSE of 0.7556 on the log scale and 21.95 on the original score scale, accompanied by an R-squared score of approximately -0.0001, which is statistically equivalent to zero. This near-zero R-squared value demonstrates that the linear combination of TF-IDF features and character length explains virtually none of the variance in question scores. This finding demonstrates that StackOverflow question scores are not determined by linguistic properties alone, but are driven by external metadata factors entirely absent from the text column, including author reputation, submission timing, topic popularity, and algorithmic thread visibility.

 47: 2.5 Task 3 — LDA Topic Modelling
 48: To discover latent thematic structures across the StackOverflow corpus, unsupervised Latent Dirichlet Allocation is performed on a representative sample of 100,000 rows. Unsupervised topic modelling requires a distinct feature extraction stage compared to supervised models. CountVectorizer is utilised in place of HashingTF because LDA requires an explicit vocabulary index with an invertible mapping to recover human-readable terms from topic-word distributions. HashingTF is a one-way hashing function with no inverse lookup, making post-hoc topic interpretation impossible. Instead of arbitrarily choosing a topic count, the number of topics was determined through a data-driven model selection step. Multiple LDA models were trained with candidate topic sizes of k in [5, 8, 10, 12] and evaluated using Log Perplexity and Log Likelihood on the vectorized question sample. The optimal model size was programmatically selected as k = 10 (the count yielding the lowest perplexity of 2.8337, representing a probabilistic elbow point). The final LDA model was trained with the selected optimal k, a maximum iteration count of 5, a vocabulary size of 5,000 terms, a minimum document frequency of 10.0, and an online variational Bayes optimiser to ensure efficient execution.
 49: 
 50: The final LDA model (k=10) completed training in a combined runtime of 198.96 seconds and yielded a log perplexity score of 2.8337 and log likelihood of -53131199.4455, where lower values indicate a tighter probabilistic fit to the observed vocabulary space.  79: 3. Discussion
 80: The multi-class classification accuracy of 71.31% achieved on a ten-class NLP problem represents a highly successful modelling outcome when placed in proper statistical context. In a balanced ten-class classification scenario, a completely random baseline model would yield a theoretical accuracy of precisely 10%. Achieving an accuracy of 71.31% therefore demonstrates that the HashingTF-IDF feature representation, combined with the ensemble of 50 decision trees, successfully extracted highly predictive linguistic signals from unstructured question text. An accuracy of this magnitude on a ten-class natural language classification problem using only bag-of-words features is consistent with published results in the programming language identification literature.
 81: 
 82: The significant variance in per-class accuracy rates reveals how domain-specific vocabulary fundamentally governs NLP classification performance. Python and iOS achieve the highest individual accuracy rates of 89.29% and 88.42% respectively. Python questions contain highly distinctive programming tokens such as numpy, pandas, pip, list comprehensions, and Python-specific indentation syntax patterns, while iOS questions are saturated with exclusive mobile development frameworks including UIKit, Swift, Xcode, storyboard, and CocoaPods. These unique lexical markers rarely appear in other programming domains, enabling the Random Forest model to establish clean decision boundaries. In contrast, JavaScript achieves the lowest per-class accuracy of only 43.82%, directly attributable to the modern web development ecosystem. JavaScript posts heavily overlap with front-end web technologies through shared HTML structural tags, CSS style selectors, DOM manipulation terminology, and jQuery library syntax. The 1,018 test case misclassification between JavaScript and jQuery is technically and semantically expected, given that jQuery is an established JavaScript library. Developers writing jQuery questions naturally incorporate vanilla JavaScript syntax, load identical browser script tags, and reference identical web browser error messages, making it linguistically impossible for a bag-of-words model to reliably separate the two classes.
 83: 
 84: The regression model's R-squared value of approximately -0.0001 represents a major substantive finding rather than a modelling failure. In statistical modelling, an R-squared value approaching zero indicates that the independent variable set fails to explain the variance of the dependent variable. This finding conclusively demonstrates that the textual quality, lexical complexity, and character length of a question are not the primary determinants of its community-voted score. This result aligns with established knowledge of StackOverflow's sociotechnical ecosystem. Community scores are driven predominantly by external social and algorithmic factors that are entirely absent from the text column: questions submitted by high-reputation users with established badges attract disproportionate views and votes regardless of textual quality; questions submitted during peak traffic windows accumulate exponentially more attention than equivalent questions submitted during off-peak hours; and questions centred on highly active, algorithmically trending technology topics receive platform-amplified visibility that dramatically inflates their score accumulation independent of linguistic merit.
 85: 
 86: The final LDA model's log perplexity of 2.8337 confirms a statistically optimised probabilistic fit, demonstrating that the 10 discovered latent topics are semantically distinct and internally coherent. The emergence of Topic: public / var / void as the corpus-dominant topic with 40,640 documents accurately reflects the real-world distributions within the sample. The relative representation of other topics reflects the specific vocabulary size configuration used.
 87: 
 88: Despite the robust statistical results obtained, the study is subject to four critical limitations that must be acknowledged. The HashingTF-IDF feature pipeline relies on a bag-of-words assumption that completely disregards word order, grammatical structure, and semantic context. A question stating this function is working, not failing is processed identically to a question stating this function is failing, not working, despite representing diametrically opposite technical scenarios. This fundamental limitation restricts the model's capacity to capture negation, conditional phrasing, and contextual nuance. Second, the assignment of human-readable labels to the ten latent LDA topics is an inherently manual and subjective interpretive process. The topic naming depends entirely upon the researcher's domain knowledge and interpretation of the highest-weighted vocabulary terms, meaning that a different analyst could reasonably assign alternative thematic labels to the same probabilistic word distributions. Third, the community score is an imperfect proxy for objective question quality. A technically precise and detailed question may retain a score of zero due to limited search traffic volume, while a common, low-effort question may accumulate hundreds of votes through historical search engine optimisation effects. Fourth, the use of stratified sampling, specifically 200,000 rows for classification and 100,000 rows for regression, to maintain local computational feasibility means the trained models may not fully generalise to the linguistic edge cases and rare technology tags present within the complete 1.26 million question corpus.

 90: 4. Conclusion
 91: This academic investigation has successfully designed and executed a comprehensive big data analytical pipeline on the StackOverflow StackSample dataset using Apache PySpark and Tableau. The five-dimensional 5 Vs analysis confirmed that this dataset satisfies all recognised characteristics of big data through its volume of over five million records, variety of five distinct data types, continuous real-time velocity of question generation, veracity concerns arising from score-biasing social factors, and significant analytical value for developer platform intelligence.
 92: 
 93: The unsupervised LDA topic model successfully extracted 10 semantically coherent programming sub-domains with a log perplexity of 2.8337, confirming that StackOverflow questions organise into distinct thematic clusters aligned with real-world programming domains. The utilisation of PySpark was essential for this project, as constructing high-dimensional TF-IDF vectors across 1.26 million rows of long-form text would cause catastrophic memory exhaustion on any single-node computing architecture.
 94: 
 95: To enhance predictive performance in future research, the bag-of-words TF-IDF pipeline should be replaced with advanced dense vector representations such as Word2Vec or transformer-based contextual embeddings including BERT and RoBERTa. These architectures preserve word order and capture contextual semantic relationships, enabling the modelling of complex programming-domain semantics that bag-of-words approaches fundamentally cannot represent. Additionally, incorporating metadata features such as author reputation score, submission timestamp, and tag historical popularity into the regression feature set would be expected to yield substantially higher R-squared values, as these factors have been identified as the primary drivers of StackOverflow score accumulation.
