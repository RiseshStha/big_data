[30] 1. Introduction
[31] In the contemporary digital landscape, data generation has scaled exponentially, necessitating the adoption of specialized distributed computing systems. This academic report investigates the StackOverflow StackSample dataset through the five-dimensional framework of big data: Volume, Variety, Velocity, Veracity, and Value (5 v’s).
[32] Table 1: The Five Vs of Big Data Applied to the Stack Overflow StackSample Dataset
[34] Processing unstructured long-form HTML text at this scale poses significant computational challenges. Standard single-node tools like Python Pandas rely on single-threaded, in-memory execution, resulting in JVM heap space exhaustion or system-wide memory failure when processing over 1.26 million text rows and generating high-dimensional TF-IDF matrices. Consequently, Apache PySpark is utilised as the distributed computing framework to enable resilient distributed datasets and partition-based parallel execution across multiple cores.
[35] This investigation pursues three primary analytical objectives: multi-class tag classification using a Random Forest classifier to predict the primary technology tag, quality score prediction using Linear Regression to examine textual influences on scores, and unsupervised Latent Dirichlet Allocation (LDA) to discover latent themes. Tableau is utilised to construct interactive dashboards representing these analytical phases. Table 2 summarises all results across the four analytical phases.

[38] 2. Implementation
[39] 2.1 Dataset Description
[40] The empirical research utilises the public StackSample dataset, a representative 10% subset of the official Stack Exchange Data Dump. Freely accessible on Kaggle (Kaggle, 2019), it contains three CSV files: Questions.csv (1,264,216 rows, 7 columns), Answers.csv (2,014,516 rows), and Tags.csv (3,750,994 rows, 2 columns). For this project, Questions.csv is inner-joined with Tags.csv to map technology tag categories to their corresponding question text.
[41] The pipeline processes five distinct data types, satisfying data variety requirements: integers (Id, Score), long-form HTML text (Body), short title strings (Title), temporal timestamps (CreationDate), and categorical tag strings (Tag).
[42] To ensure clean numerical inputs for machine learning, preprocessing is executed sequentially in PySpark: 1) HTML tags are removed using regular expressions (<[^>]+>); 2) Null value filtering confirms complete records in core columns (Title, Body, Score); 3) Title and Body are concatenated into a unified text column; 4) A logarithmic transformation (log1p) is applied to compress the highly right-skewed score distribution; 5) Multiple tag rows are aggregated and inner-joined to the questions schema.
[43] 2.2 Environment and Configuration
[44] The distributed environment is established on Windows 11 using Apache Spark 3.5.8 and Python 3.10. The SparkSession is configured with a 4GB driver memory allocation and 50–100 shuffle partitions optimised for local parallel execution.
[45] Apache Spark operates on a master-worker driver-executor architecture. The driver program coordinates application flow, schedules tasks, and compiles directed acyclic graphs. Independent executor processes perform partition-level operations in parallel. This distributed in-memory computation is essential: constructing high-dimensional TF-IDF vectors across 1.26 million rows would exhaust physical memory on single-node systems. PySpark bypasses this by partitioning data blocks across processor cores.
[46] 2.3 Task 1: Tag Classification (Random Forest)
[47] The classification workflow is implemented via a six-stage NLP pipeline in PySpark MLlib: Tokenizer segments text into words; StopWordsRemover filters common prepositions and pronouns; HashingTF maps tokens into a 32,768-dimensional sparse term-frequency vector; IDF scales features to penalise ubiquitous terms; StringIndexer encodes categorical tag targets; and a Random Forest Classifier (50 decision trees, depth 10, subsampling rate 0.8) trains on the feature vectors.
[48] To prevent class imbalance and maintain computational feasibility, stratified sampling extracts 20,000 records per tag class. The balanced dataset of 200,074 rows is split into an 80% training set (160,236 rows) and a 20% test set (39,838 rows) across the top ten technology tags.
[49] Random Forest is preferred over multinomial Logistic Regression because it handles sparse high-dimensional TF-IDF vectors without feature scaling or regularisation tuning, supports multi-class classification natively, resists overfitting, and generates feature importance scores for post-hoc interpretability.
[50] The model achieved an overall accuracy of 71.31%, F1-score of 0.7076, weighted precision of 0.7204, and recall of 0.7131, completing training in 6 minutes and 35 seconds. Table 3 presents the accuracy across all ten tag classes.

[60] 2.4 Task 2: Score Prediction (Linear Regression)
[61] The regression pipeline predicts question scores based on textual properties. It mirrors the classification pipeline's NLP phases, employing tokenization, stop word removal, HashingTF (16,384 features), and IDF scaling. A VectorAssembler combines the sparse TF-IDF vector with a numerical text_length feature. This assembly feeds into a Linear Regression estimator (maximum 10 iterations, regularisation 0.1, ElasticNet mixing 0.5 to balance L1 and L2 penalties).
[62] The target variable (Score) is heavily skewed (mean 1.78, range -73 to 5,190). A log1p transformation compresses the distribution and stabilizes residual variance. The model is trained on a stratified sample of 100,000 questions (Score ≥ 0), split into an 80/20 train/test partition.
[63] Linear Regression is selected to establish an interpretable baseline for textual influence on community votes. The model achieved an RMSE of 0.7556 on the log scale (21.95 original) and an R-squared of -0.0001 (statistically equivalent to zero). This near-zero R-squared indicates that text features and question length explain virtually no score variance. Question scores are driven by external factors such as author reputation, submission timing, and algorithmic visibility rather than linguistic attributes.

[66] 2.5 Task 3: LDA Topic Modelling
[67] To discover latent topics, unsupervised LDA is applied to 100,000 question samples. CountVectorizer is used instead of HashingTF because LDA requires an invertible vocabulary index to map terms back to topics. The optimal number of topics was programmatically determined as k=10 by training candidate models (k in [5, 8, 10, 12]) and finding the perplexity elbow point. The final LDA model was trained with k=10, 5 max iterations, vocabulary size of 5,000, minimum document frequency of 10.0, and an online variational Bayes optimiser.
[68] The optimal model completed training in 198.96 seconds, yielding a log perplexity of 2.8337 and log likelihood of -53,131,199.45. Table 4 details the discovered topics, their document distribution counts, and representative top vocabulary terms.

[96] 3. Discussion
[97] The classification accuracy of 71.31% on a ten-class NLP problem represents a highly successful outcome. Compared to a random baseline of 10%, the Random Forest classifier successfully extracts strong predictive text signals from unstructured questions using basic bag-of-words features.
[98] Class accuracy varies significantly based on vocabulary uniqueness. Python and iOS achieve the highest accuracy (89.29% and 88.42%) due to distinct tokens (e.g., numpy, pandas, UIKit, Swift, storyboard) that rarely appear in other domains. Conversely, JavaScript achieves the lowest accuracy (43.82%) due to massive vocabulary overlapping with HTML, CSS, and jQuery. The 1,018 test-set misclassifications between JavaScript and jQuery are semantically expected, as developers discussing jQuery natively use JavaScript code and share web browser error environments.
[99] The near-zero R-squared value (-0.0001) confirms that question scores are not determined by linguistic properties or text length. This aligns with StackOverflow's sociotechnical dynamics: scores are driven by external factors such as author reputation, peak submission timing windows, topic popularity, and algorithmic thread visibility.
[100] The final LDA model's log perplexity of 2.8337 demonstrates semantically distinct and coherent topic separation. The emergence of Topic: public / var / void as the dominant thematic cluster (40,640 documents) accurately maps to the high volume of structural debugging queries in the corpus.
[101] The study has four limitations: 1) The TF-IDF pipeline uses a bag-of-words model that disregards syntax and word order; 2) Subjective human interpretation is required to assign semantic labels to LDA topics; 3) Community score is an imperfect proxy for question quality due to SEO and traffic variance; 4) Stratified sampling constraints (200,000 and 100,000 rows) limit the models' exposure to rare edge cases in the full dataset.
[102] 4. Conclusion
[103] This investigation executed a multi-task big data analytical pipeline on the StackOverflow dataset. The 5 Vs analysis confirmed the big data characteristics of the corpus, and PySpark was demonstrated as essential to prevent memory exhaustion during high-dimensional NLP feature engineering across 1.26 million rows.
[104] The PySpark models successfully achieved 71.31% classification accuracy and extracted 10 coherent LDA topics (log perplexity 2.8337), while regression confirmed that question scores are independent of textual quality.
[105] Future work should replace the bag-of-words pipeline with dense neural architectures like Word2Vec or transformer embeddings (BERT) to capture word order. Additionally, integrating metadata features (author reputation, timestamps) would improve regression score predictions.
